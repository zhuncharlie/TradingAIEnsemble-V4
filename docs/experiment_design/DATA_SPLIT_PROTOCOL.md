# DATA_SPLIT_PROTOCOL.md

Status: design document. No adapters, schema, harness, or Python code are
modified by this document or its adoption. All windows/splits below are
proposals for the harness/experiment-execution layer to implement; nothing
here overrides `CONTRACT/schemas.py` or `harness/q4_protocol.py` causality
guarantees — it relies on them.

Companion documents: `BASELINE_DESIGN.md`, `METRIC_DESIGN.md`,
`EXPERIMENT_PROTOCOL.md` (master document, references this one for every
experiment's data window).

---

## 0. Why a naive single time-based split does not work here

A standard "80/10/10 chronological split" assumes one dataset generated by
one process. This project has 26 heterogeneous adapters wrapping
independent upstream projects, and empirically (`results/unified_harness/`,
run 2026-07-18, `as_of=2024-01-15`, universe `AAPL/MSFT/NVDA`) they differ on
two axes that a single split cannot paper over:

1. **Replay capability.** Some adapters compute decisions purely from
   price/factor history and accept an arbitrary historical `as_of` (they can
   be re-run against any past date without fabricating anything). Others
   depend on live, current-context information (news feed, LLM call against
   "today") that cannot be honestly regenerated for a historical date
   without inventing the historical information context — doing so would
   violate CLAUDE.md §2 ("do not fabricate ... evidence").
2. **Native operating window.** Several Q4 adapters are hard-bound to a
   fixed upstream dataset/era, not a free choice of the harness. Observed
   directly in the unified-harness run's `gen_start`/`gen_end` per adapter:
   `earnmore` → `2008-09-01..2009-03-01` (bundled crisis-era dataset),
   `trademaster` → `2021-01-04..2021-12-31` (fixed DJ30 dataset), `skfolio`
   → `2022-01-01..`, `finrl` → `2022-07-01..`. These are not tunable by this
   harness; the split protocol must accommodate them, not assume them away.

Consequently this protocol defines **two adapter classes** and **windows
scoped per class**, plus one **shared recent window** used whenever adapters
of both classes must be compared side by side.

## 1. Adapter replay classification

| Class | Definition | Adapters (26 total) |
|---|---|---|
| **R — Replay-capable** | Decision is a deterministic-or-stochastic function of point-in-time price/factor/fundamental data; upstream code accepts an arbitrary historical `as_of`/`gen_start`/`gen_end` without depending on "current" news or a live LLM call about today. Confirmed STEPWISE or STATIC_ONLY causal execution per `q4_stepwise_support.csv`. | `deepalpha`, `alphagen`, `atlas`**†**, `alphaforge`, `qlib`, `rdagent`, `finrl`, `finrl_x`, `deepdow`, `earnmore`, `trademaster`, `skfolio`, `universal_portfolios`, `pgportfolio`, `finclaw`, `vibe_trading` (16) |

**† `atlas` caveat (found during self-review, `docs/research_reports/
2026-07-19_experiment_protocol_self_review.md` finding C2):** `atlas`'s
bundled dataset is crypto-perpetuals only (`2021-11-06..2024-08-14`), per
the known-limitation note already recorded in `results/observations/
observation_batch_day1_historical_extension/index.csv`. For any
non-crypto ticker (i.e. every ticker in the §2.1 universe below except a
crypto pair), `atlas` silently substitutes the *same BTCUSDT signal* under
the requested ticker's label (self-disclosed by the adapter in its own
`supporting_evidence`, so this is upstream behavior being faithfully
surfaced, not a harness-side fabrication). Consequence: `atlas` is Class-R
by execution mechanics, but its Q3 output across the standard 10-ticker
equity/ETF universe is **not 10 independent signals** — it is one signal
repeated 10 times under different labels. Any experiment that would treat
`atlas`'s per-ticker outputs as independent observations (e.g. D1 field
coverage counts, D4 contradiction counts, M1-M4 fusion inputs) must either
exclude `atlas` from non-crypto-ticker analysis or explicitly collapse its
10 "observations" to 1 before counting — do not let it inflate N by 10×.
| **L — Live/current-context-bound** | Decision genuinely depends on real-time information (news headlines, live sentiment, an LLM asked to reason about "today", forecasting-market consensus) that cannot be faithfully backfilled to a historical date without fabricating the historical information context. | `ai_hedge_fund`, `tradingagents`, `finmem`, `finagent`, `finrobot`, `fingpt`, `finbert`, `prediction_arena`, `quantmuse`, `agentictrading` (10) |

Notes:
- `finbert` is Class L not because its model is time-dependent but because
  its adapter's headline source (`yfinance` current headlines) only exposes
  *current* headlines, not historical-as-of headlines — documented already
  in `reports/icaif_experiments/calibration_README.md`. If a point-in-time
  news archive is ever wired in, `finbert` should be promoted to Class R;
  until then, treating it as R would silently violate causality by pairing
  today's headline with a historical decision date.
- `agentictrading` is Class L operationally (STATIC_ONLY, one real
  leaderboard-snapshot call, no historical replay mechanism upstream) even
  though it carries no live-news dependency — grouped with L because it
  cannot be re-run against arbitrary past dates either.
- `vibe_trading` is Class R via `LEGACY_INTERNAL_LOOP` replay only (its
  vectorized backtest is computed once and replayed one decision per harness
  step) — flag every `vibe_trading` result with this caveat, per
  `q4_stepwise_support.csv`.
- This table must be re-derived, not assumed, whenever a new adapter is
  added — add one row to `adapter_capability_recovery.csv`'s successor
  document, do not silently edit this table without re-verifying against
  the adapter's actual source (per CLAUDE.md §2, no fabricated
  classification).

This classification is the single most load-bearing decision in this
protocol: every window below is defined per-class, and any experiment that
mixes classes (most Layer 2 fusion experiments do) must explicitly say
which comparison it is making — same-class apples-to-apples, or
knowingly-mixed with the caveat stated in the write-up.

## 2. Two data planes, evaluated separately

### 2.1 Plane A — realized market data (for scoring, not for adapter input)

Used only to compute forward returns / calibration / portfolio P&L after a
decision has already been recorded. Never fed to an adapter as if it were
available before its `data_cutoff`.

- **Source**: `yfinance`, already the project's price provider
  (`analysis/icaif_data_loader.py::YFinanceFutureReturnProvider`,
  `analysis/fetch_price_history.py`). Verified during this design pass:
  `yf.Ticker("AAPL").history(period="10y")` returns 2,514 real daily bars
  from 2016-07-18 to 2026-07-17 — **10 years of history is available now**
  for the project's existing liquid-ticker universe, no new credential
  needed.
- **massive.com MCP data availability — checked and currently insufficient
  for this purpose.** `mcp__massive__call_api` against
  `/v2/aggs/ticker/AAPL/range/1/day/2016-07-19/2026-07-19` returned real
  bars starting only at **2024-07-19** (~2 years back); an explicit probe
  for `2015-01-01..2015-12-31` returned `HTTP 403 NOT_AUTHORIZED — "Your
  plan doesn't include this data timeframe"`. The MCP server itself is
  correctly configured; the *plan tier* behind it caps historical daily
  aggregates at ~2 years. **Recommendation to the user**: keep `yfinance`
  as the Plane-A provider for the 10-year backtest windows below — it
  already covers what's needed for equities/ETFs/major crypto pairs used in
  this project. Upgrading the massive.com plan would still be worth doing
  if/when this project needs data `yfinance` cannot provide (options chains,
  intraday microstructure, point-in-time news/filings for Class-L causality,
  reference/fundamentals) — that is a different, complementary need, not a
  substitute for the price history already available.
- **Universe**: reuse the pilot's 10-ticker + 2 diversification asset
  universe for continuity — `NVDA, AAPL, MSFT, TSLA, SPY, QQQ, JPM, XOM,
  JNJ, GLD` (single-asset Q1/Q2/Q3) `+ TLT, CASH` (Q4 portfolio scope).
  Extending the universe is allowed but must be justified per experiment
  (e.g. Ro3's event stress test may need pre-2016 tickers for the 2008 GFC
  slice, scoped to Class-R adapters that actually reach that far, i.e.
  `earnmore` only as observed).

### 2.2 Plane B — adapter decisions (the thing under evaluation)

Must be generated by actually invoking each adapter (per
`CONTRACT/adapter_runner.py`) at a chosen `as_of`, never synthesized,
interpolated, or copied across adapters. Plane B is class-scoped:

- **Class R**: can be walk-forward generated across the full window in
  §3.1, subject to each adapter's own hard-bound sub-window (see §1 table;
  `trademaster`/`earnmore`/`skfolio`/`finrl` already show these bounds).
- **Class L**: can only be generated **prospectively from the protocol's
  adoption date forward** (see §3.2) — there is no honest way to backfill
  10 years of Class-L decisions without fabricating historical news/LLM
  context, which CLAUDE.md §2 forbids outright.

## 3. Split definitions

### 3.1 Class R — full historical walk-forward split

```
|---------------- TRAIN (adapter-internal) ----------------|--CAL--|--VAL--|====TEST====|
2016 ......................................................2023-07..2024-07..2025-07....2026-07
```

- **TRAIN**: whatever window each upstream project already used to fit its
  own model/policy (e.g. `qlib`'s `LGBModel`, `deepdow`'s `GreatNet`,
  `trademaster`'s `EIIEConv`). This harness does not retrain upstream
  models and does not choose this window — it is documented per-adapter
  from `Q4Policy.generation_window` / `RunMetadata` and reported, not
  re-used for evaluation. This is what CLAUDE.md §4 calls "native upstream
  output" — preserved, not overwritten.
- **CALIBRATION (CAL)**: last ~12 months before the validation window
  (nominally 2023-07 to 2024-07, adjustable to each adapter's actual
  reachable window). Used to: (a) fit Layer 1 reliability/calibration
  profiles per adapter (D2 in `EXPERIMENT_PROTOCOL.md`), (b) fit any
  learned Layer 2 fusion/routing parameters (M2–M4). Re-runnable, not
  frozen.
- **VALIDATION (VAL)**: the following ~12 months (nominally 2024-07 to
  2025-07). Used only for: threshold selection, baseline-vs-method
  comparison during method development, early ablation iteration. May be
  re-run multiple times as methods are refined — this is the "tuning"
  surface, VAL absorbs iteration so TEST doesn't have to.
- **TEST**: most recent ~12 months (nominally 2025-07 to 2026-07),
  **embargoed**: touched at most once, after all methods/thresholds are
  frozen from VAL. No re-tuning against TEST results is permitted — a
  method that fails on TEST is reported as failing, not iterated against
  TEST until it passes (that would be test-tuning, explicitly forbidden by
  the task brief).
  **Already-contaminated dates — verified, must be excluded from TEST
  (self-review finding C1):** Class-R adapters `deepalpha`, `alphagen`,
  `atlas`, `finrl`, `qlib` already have real, already-analyzed decisions
  recorded at `2026-05-15, 2026-05-21, 2026-05-27, 2026-06-02, 2026-06-08`
  (`results/observations/observation_batch_day1_historical_extension/`)
  and `2026-07-06` (`results/comparison_2026-07-02/` main batch), all of
  which fall inside the nominal `2025-07..2026-07` TEST window — this
  design pass itself read and summarized that data. These dates (plus
  `2026-07-18`, the `unified_harness` run's collection date) are
  **excluded from the Class-R TEST window** and remain CAL-equivalent
  material only; they must never be reported as TEST-window results for
  any Class-R adapter. The `unified_harness_2026_07_18` run's own
  `as_of=2024-01-15` decision date is unaffected — it falls inside CAL,
  not TEST, and is unrestricted.
- **Purge/embargo gaps**: insert a gap of `max(horizons) = 20 trading days`
  between CAL/VAL and VAL/TEST boundaries so no forward-return label at the
  20-day horizon computed near a boundary leaks information across it
  (standard walk-forward purging, needed because Plane-A forward returns at
  a decision near the boundary reach past it).
- **Per-adapter reachability caveat**: several Class-R adapters cannot
  reach the full 2016–2026 span (see §1). For those, CAL/VAL/TEST are
  proportionally compressed within the adapter's own reachable window and
  the compression is reported per-adapter, not silently padded with
  fabricated decisions.

### 3.2 Class L — prospective accumulation split

Class L cannot have a historical TRAIN/CAL/VAL/TEST split at all without
fabricating history. Instead:

```
Protocol adoption date (T0) -----> ongoing daily/weekly snapshot collection
        |---------- accumulation window (CAL-equivalent) ---------|--TEST--|
```

- **T0** = the date this protocol is formally adopted for live collection
  (recommend: first daily run after `EXPERIMENT_PROTOCOL.md` is approved).
- **Accumulation window**: every snapshot collected from T0 onward feeds
  Layer 1 diagnostics (D1–D7) and Layer 2 calibration-equivalent fitting,
  exactly like Class R's CAL/VAL, just accumulated going forward instead of
  sliced from history.
- **TEST**: a forward-declared future window (e.g. "the 60 days starting
  90 days after T0") — declared *before* it starts, not chosen after seeing
  results, so it cannot be cherry-picked.
- **Consequence**: any Layer 2 method that mixes Class R and Class L
  adapters (most realistic fusion methods do — this is the whole point of
  the ensemble) is bound by the *slower* class. Its TEST window is Class
  L's forward-declared window, not Class R's 2025-07..2026-07 slice, unless
  the experiment explicitly restricts itself to Class-R-only adapters (in
  which case say so and use §3.1's TEST window instead).
- This directly reuses and extends the pilot's existing snapshot batches
  (`results/comparison_2026-07-02/`, `results/observations/
  observation_batch_day1{,_historical_extension}/`, `results/
  unified_harness/unified_harness_2026_07_18/`) as the *start* of the
  accumulation window, not as a pre-existing TEST set — none of that data
  has been embargoed or held out, so none of it may be reported as a test
  result; it is CAL-equivalent material only.

### 3.3 Shared recent window (cross-class comparability)

For any experiment that must compare Class R and Class L adapters on
literally the same decision dates (e.g. D5 regime profiling across *all*
26 adapters, D4 cross-adapter contradiction detection), use only the
**shared recent window**: the intersection of Class R's reachable dates and
Class L's accumulation window. Concretely today that is small — the pilot's
2026-05-15 through 2026-07-18 snapshots plus whatever accumulates after T0.
This window grows over time; experiments that depend on it (D4, D5, M3)
should be re-run periodically as it grows rather than run once on a
too-small sample, echoing the pilot's own limitation note ("39 calibration
buckets, 28 with n<10").

## 4. Market regime and event segmentation

### 4.1 Regime tagging (feeds D5, Ro1)

Tag each trading day in Plane A with a regime label derived from the
benchmark (`SPY`) alone — a fixed, documented, non-adapter-derived rule so
regime labels can't be accused of circularity with any adapter's own Q2
`market_regime` state estimate (that comparison is itself part of D5/D8's
coherence check, so the ground-truth regime tag must be independent):

- **Trend**: sign and magnitude of trailing 60-trading-day return.
- **Volatility**: trailing 20-day realized volatility vs. its own trailing
  252-day distribution (percentile bucket: low/mid/high).
- **Regime label** = a simple cross of the two: `BULL_LOWVOL`,
  `BULL_HIGHVOL`, `BEAR_LOWVOL`, `BEAR_HIGHVOL`, `SIDEWAYS` (trend near
  zero, either vol bucket collapsed to one `SIDEWAYS` label to avoid
  over-fragmenting small samples).
- This is a labeling convention for slicing evaluation results, **not** a
  new adapter capability and not written into `CONTRACT/`.

### 4.2 Historical event stress windows (feeds Ro3)

Declared in advance, not chosen post-hoc from favorable results:

| Event | Window | Reachable by |
|---|---|---|
| 2008 Global Financial Crisis | 2008-09-01 to 2009-03-01 | `earnmore` only (confirmed reachable — this is literally the window it was run against in `unified_harness_2026_07_18`); any other Class-R adapter must be verified reachable before inclusion, not assumed |
| 2020 COVID crash | 2020-02-15 to 2020-04-30 | Class-R adapters whose `TRAIN` window predates 2020 and whose upstream data source covers it — verify per adapter, do not assume |
| 2022 rate-hike bear market | 2022-01-01 to 2022-10-15 | Most Class-R adapters (several already show `gen_start` at or before 2022) |
| Contemporary event (TBD) | first major single-day move ≥3σ on `SPY` occurring inside the Class-L accumulation window after T0 | All adapters (first genuinely shared-live stress event) |

Ro3 is explicitly exploratory (small N per event) — see
`EXPERIMENT_PROTOCOL.md`'s failure-interpretation note for Ro3: a null or
inconsistent result here is expected and must not be treated as
disconfirming, only as "insufficient event count," per the same honesty
standard the pilot applied to `insufficient_data`.

## 5. Anti-leakage checklist

This protocol relies on, and does not replace, causality mechanisms already
enforced by the codebase — verify these remain true, do not re-implement
them:

- `QueryContext.data_cutoff <= as_of` and `PolicyDecisionStep.
  information_cutoff <= timestamp` — enforced at the schema/test layer
  (`tests/test_point_in_time_guards.py`, `tests/test_q4_causality.py`,
  `tests/test_q4_step_protocol.py`). This protocol's job is to choose
  windows that respect these guarantees, not to check them itself.
- **No test-window generation until CAL/VAL work is frozen.** Practically:
  do not call any adapter with an `as_of` inside a TEST window until the
  experiment's method/threshold choices are committed. If a Class-R
  adapter is re-run and happens to touch the TEST window as a side effect
  (e.g. a wide `gen_end`), that run's TEST-window output must be embargoed
  from result files used for method selection, not merely "not looked at."
- **Forward-return purge gaps** (§3.1) applied consistently across every
  experiment that computes 1/5/20-day forward returns near a split
  boundary — reuses the existing `FutureReturnProvider` contract (never
  fabricates a return before enough trading days have elapsed).
- **No re-use of pilot "future" claims.** The pilot's `PAPER_FINDINGS.md`
  and `calibration_README.md` numbers were computed against a
  now-superseded v1/Q5 schema and a 15-adapter set; they are informative
  precedent (cited in `EXPERIMENT_PROTOCOL.md`) but are not pre-existing
  TEST-window results for the v2/26-adapter protocol defined here.
- **Multi-agent document safety**: per the session's standing instruction,
  any subagent producing supporting analysis for this split (e.g. a fork
  computing actual per-adapter reachable-date bounds from a full 26-adapter
  sweep) writes its findings to `docs/research_reports/`, never edits this
  file directly; only the coordinator updates
  `docs/experiment_design/DATA_SPLIT_PROTOCOL.md` itself.

## 6. Open items / explicitly deferred

- Exact CAL/VAL/TEST calendar boundaries (§3.1) are nominal and should be
  finalized once a full 26-adapter reachable-window sweep is run (D1 in
  `EXPERIMENT_PROTOCOL.md` produces this as a byproduct) — do not hand-wave
  past this before Layer 1 diagnostics start.
- T0 for Class L (§3.2) is not yet set — it is a scheduling decision for
  whoever runs the harness, not a data-availability question, so it is left
  to the experiment-execution owner rather than fixed here.
- If the user upgrades the massive.com plan, re-verify Plane A provider
  choice — `yfinance` remains adequate for equities/ETFs, but massive.com
  may become the more suitable Plane-A/B-adjacent source for options,
  point-in-time news (which would let `finbert`/Class-L causality actually
  work historically), or reference/fundamentals data this project does not
  yet have a source for.
